带你见识不一样的Java异常处理

发表于 2017-05-23

最近在复习JAVA基础，发现对Java异常处理感觉很陌生，只是知道一些基本的用法所有就觉得应该做一个总结，就有了这篇文章。

主要内容

1. 为什么会有异常处理机制
2. 异常分类
3. 如何声明和抛出异常
4. 如何捕获异常

为什么会有异常处理机制

    在理想状态下我们的程序是没有任何的错误，但是现实并不是如此。就像用户输入数据的格式不一定正确，选择的文件有可能不存在，服务器也不可能一直保持一个健康的状态，这些都是程序中可能出问题的点。其中一些错误我们程序员解决不了的，比如硬件问题或者是服务器内部出现问题，这些我们统称是Error（错误），还有一种是我们程序员可以解决的例如：数组越界、空指针等，这些问题统称为Exception(异常)。

    既然这些问题可以被解决，而且解决这问题的方法可能有很多种，那为什么不制定一套规范那？于是在Java中就有了一整套异常处理机制，这套规范给出了对于这种问题完整的解决方法。大致来讲就是将控制权从错误产生的地方转移给能够处理这种情况的错误处理器中，能使用户返回到一种安全的状态并能完成一些其他操作或者是保存用户之前所有操作的结果，然后以适当的方式终止程序。

异常分类

    从图中我们可以看出所有的异常对象都是派生于Throwadble类的一个实例，这向下是就是我们在前面提到的Error（错误）和Exception(异常)。其中Exception是今天研究的重点。

Error

    这个类描述了Java运行时系统的内部错误和资源耗尽错误。在应用程序中不应该抛出这个类型的对象。只能告知用户，并尽力使程序安全终止。

Exception

    在这个类下面有分为RuntimeException（运行时异常）和IOException（其他异常）。 这两类的分类条件为：由 程序错误 导致的异常属于RuntimeException;而程序本身没有错误，但是出现了像I/O错误这类问题导致的异常属于其他异常。从图中可以看出RuntimeException类的子类都是一些我们平时比较常见的异常，也就是说这类异常是我们可以在程序避免出现的问题。

我觉得有一句话就很好的说明了这个问题： 

如果出现RuntimeException异常，那么就一定是你的问题。

    在Java语言规范中将Error和RuntimeException类所产生的的异常称为未检查异常，所有的其他异常称为已检查异常，并且编译器也会检查是否为所有的已检查异常提供异常处理器，

如何声明和抛出异常

既然前面已经介绍了异常的分类了，接下来我们就来了解如何声明和抛出一个异常。

声明一个已检查异常

public void test() throws FileNotFoundException

    这个例子就很好的展示了我们在程序中如何声明一个异常。我们应该在方法首部声明所有可能抛出的异常。但是有的时候我们也不必将所有的异常都在方法首部声明，但是在下面这四种情况下一定会抛出异常。

1. 调用一个抛出已检查异常的方法，例如，FileInputStream构造器。
2. 程序运行过程中发现错误，并且利用throw语句抛出了一个已检查异常。
3. 程序出现错误，例如，数组越界。
4. Java虚拟机内部错误。

    如果遇到前两种情况之一，那就必须在方法首部声明有可能抛出的异常，因为如果没有处理器捕获这个异常，当前执行的线程就会结束。对于那些有可能被其他人使用的Java方法，应该根据异常规范，在方法首部声明这个可能抛出的异常。

class MyTest{
    ...
    public Image loadImage(String s) throws IOException{
        ...
    }
}

    如果在一个方法中有可能抛出多个已检查异常，也必须在方法首部列出所有可能的异常，每个类之间用逗号隔开。

class MyTest{
    ...
    public Image loadImage(String s) throws FileNotFoundException,EFOException{
        ...
    }
}

    如果是后两种情况的话，则不需要再方法首部声明异常。为什么不需要声明那？首先，从Error类继承的异常类，我们的代码是具有声明这些异常的能力，但是我们无法控制这些代码。还有就是RuntimeException，这类异常在程序编写的时候是可以避免的，我们应该去花上一定的时间去找出并解决这类问题。

###如何抛出异常###
    我们在程序中使用throw关键字来抛出异常或者是创建一个异常对象然后使用throw来抛出这个异常。

throw new IOException

IOException e = new IOException();
throw e;

如何捕获异常

    捕获异常时一个非常重要的环节，因为如果某个异常发生的时候没有在任何的地方对其进行捕获，那么程序就会终止，并在控制台上打印出异常语句，这种情况是在开发的时候可能不会造成多大的影响，但是一旦项目上线后出现这种情况对于用户的体验是非常差的。所以我们要在程序中捕获异常。

    我们在Java中使用try/catch语句块来捕获异常。

try
{
    ...
    ...
    ...
}catch(Exception e){
    ...
}

捕获多个异常

    捕获多个异常有两种方式：

   1).为每个异常提供单独的catch语句

try
{
    // 代码中抛出异常
}
catch(FileNotFoundException e)
{
    // 处理异常
}
catch(UnknownHostException e)
{
    // 处理异常
}
catch(IOException e)
{
    // 处理异常
}

   2).使用合并的catch语句

try
{
    // 代码中抛出异常
}
catch(FileNotFoundException | UnknownHostException e)
{
    // 处理异常
}
catch(IOException e)
{
    // 处理异常
}

   这种方法只能在当前捕获的异常彼此之间不存在子类关系的时候才能使用。而且这些异常变量隐含为final变量，即不能该变量赋予不同的值。

finally子句

    finally子句是异常处理中非常重要的一个子句，作为笔试题中的常客，也是比较容易出错的一个点。
在这里总结几个比较重要的点：

1. 不论是否有异常被捕获，finally子句都会执行。
2. 如果在try语句块中使用return语句退出，在方法返回前finally语句将会执行，若finally语句块中有return语句，则这个语句的返回值将覆盖原始语句的返回值。
3. 如果在try语句块中抛出了异常，而finally语句中也抛出了异常，这种情况下原始的异常将会丢失，转而抛出finally语句中的异常。

try/catch语句块的执行情况

try
{    
    // 1
    代码中抛出异常
    // 2
}
catch(IOException e)
{
    // 3
    处理异常
    // 4
}
finally
{
    // 5 相关代码
}
// 6 其他代码

我们分三种情况来分析执行情况：

  1. 代码没有抛出异常。这种情况下先执行try语句块中的全部语句，然后因为没有异常所以不执行catch语句块，然后执行finally语句块，最后执行程序中的其他代码，这里是执行1、2、5、6语句。
  2. 抛出一个在catch语句中捕获的异常。在这种情况下，程序会先执行try中语句，直到发生异常为止，然后跳过剩余的语句执行catch中匹配的语句，最后执行finally中的语句，这里是执行1、3、4、5、6语句。如果catch中抛出了异常，这个异常将被抛回给方法的调用者，这里是执行1、3、5语句。
  3. 代码抛出一个异常，但这个异常不是由catch捕获。这种情况下会先执行try中的语句，直到发生异常为止。此时将跳过try中剩余的语句，去执行finally中的语句。然后将这个异常抛给这个方法的调用者。这里讲执行1、5语句。

总结

    异常处理作为Java中相比于其他的知识点来说还是比较简单的，希望各位能通过我这篇学到一些东西。

Servlet工作原理解析

发表于 2017-05-22

JAVA WEB 技术是当今主流的互联网WEB应用技术之一，而Servlet则是JAVA WEB 中的核心基础。最近在看了《深入分析Java Web技术内幕》(许令波 著)这本书发现了以前没有看到的内容，就想对书中的内容进行总结，也方便自己以后的回顾，所以就有了这篇文章。

通过这篇文章你将了解到以下内容：

1.Servlet容器是如何工作
2.web应用中的Servlet如何创建
3.Servlet是如何被调用

Servlet容器的启动

在Tomcat中整个的体系结构中，Context容器直接管理Servlet在容器中的包装类Wrapper，所以Context是真正管理Servlet的容器。通过图我们还可以看出，一个Context对应着一个Web工程。在Tomcat配置文件中就可以看出：

<Context path="/projectOne " docBase="D:\projects\projectOne" reloadable="true" />

既然Context容器如此的重要我们就来了解下Tomcat是如何解析Context容器的。

首先，我们已经知道了Context容器是Tomcat运行时的容器，它只有在添加web应用是被加载。

  public Context addWebapp(Host host, String url, String path) { 
   silence(url); 
   Context ctx = new StandardContext(); 
   ctx.setPath( url ); 
   ctx.setDocBase(path); 
   if (defaultRealm == null) { 
       initSimpleAuth(); 
   } 
   ctx.setRealm(defaultRealm); 
   ctx.addLifecycleListener(new DefaultWebXmlListener()); 
   ContextConfig ctxCfg = new ContextConfig(); 
   ctx.addLifecycleListener(ctxCfg); 
   ctxCfg.setDefaultWebXml("org/apache/catalin/startup/NO_DEFAULT_XML"); 
   if (host == null) { 
       getHost().addChild(ctx); 
   } else { 
       host.addChild(ctx); 
   } 
   return ctx; 
}

当一个WEB应用被添加时Tomcat将会创建一个StandardContext容器，并给这个Context容器设置必要的参数，URL和path分别代表这个应用在访问路径和这个应用实际的物理路径。其中最重要的配置就是ContextConfig,这个类非常重要，是负责整个WEB应用的解析工作。

我们就来看看这个ContextConfig：

ContextConfig继承了LifecycleListener接口，它是在调用addWebApp方法时被加入到StandardContext容器中的，负责整个web应用的配置文件解析工作。
自带的init()方法和startInternal()方法最为重要
1.init()主要完成以下工作
    -创建用于解析的XML配置文件的contextDisester对象。
    -读取默认的context.xml配置文件，如果存在则解析。
    -读取默认的Host配置文件，如果存在则解析。
    -读取默认的Context自身的配置文件，如果存在则解析。
    -设置Context的DocBase。
2. startInternal()方法
    -创建读取资源文件的对象
    -创建ClassLoader对象。
    -设置应用的工作目录。
    -启动相关的辅助类。
    -修改启动状态，通知感兴趣的观察者。
    -子容器的初始化。
    -获取ServletContext并设置必要的参数
    -初始化“load on startup”的Servlet。

最后将这个Context容器添加到父容器中，然后就是调用Tomcat的start方法启动Tomcat。这样就完成了整个WEB应用加载的前期工作。

WEB应用的初始化

学过Javaweb基础的同学应该知道，web.xml是我们整个web工程中不可缺少的一个部分，如果你的项目中缺少了它，可能就会导致一些
必要的BUG。同时，整个web应用的初始化主要也是解析这个文件，因为这个文件描述了你的web项目中一些关键信息，也是整个web项目的入口。

接下来我们来看下Tomcat是如何加载到这个如此重要的配置文件的。

1.首先会找到globalWebXml
2.接着会找到hostWebXml
3.最后是寻找应用的配置文件web.xml

完成以上的步骤之后，Tomcat会将web.xml中的相应属性保存到WebXml对象中。现在很多的后台web应用都已经升级到Servlet3.0了，如果你的项目也支持了这个版本，那么在解析配置文件的时候，还要完成Servlet3.0中新增的特性的解析以及对annotat的支持。Servlet3.0在这里就不多介绍了，他不是本文的重点内容。

将属性存入WebXml对象后，也将相关的属性设置到Context容器中，这些属性包括Servlet,listener,filter,其中Servlet被包装成具有容器属性的StandardWrapper。

看到这里我们来对web.xml的作用进行一个总结

1.web应用的解析入口
2.作为用来指定context容器属性的配置文件

Servlet的创建与初始化

首先我们来看看Tomcat是如何创建Servlet：
创建Servlet实例的方法是从Wrapper.loadServlet开始的，这个方法要完成的就是获取servletClass,然后交给InstanceManager去创建一个基于servletClass.class的对象。

如果在web.xml中配置了 。。。。这个参数的话，在创建Servlet时初始化的就不是你自己编写的JAVA类了，而是conf/web.xml中的org.apache.jasper.servlet.JspServlet。

Servlet的初始化操作是在StandardWrapper的initServlet方法中完成的，作用就是调用Servlet的init(),同时将StandardWrapperFacade作为ServletConfig传给Servlet。

StandardWrapperFacade的作用就是从StandardWrapper中拿到的数据只是ServltConfig中规定的数据，而不把不关心的数据暴露给Servlet，起到了对数据封装的作用。

如果初始化的是JspServlet，那Tomcat会模拟一次请求，去请求这个JSP文件，为的是将这个jsp文件编译成类，并初始化这个类，用于后面的使用。

Servlet如何工作

在前面的内容中我们了解了Servlet的加载、创建、初始化，下面的内容中我们就来看看Servlet是如何被调用的。

用户向服务器发送请求通常包含的信息：

http://hosthome:port/contextpath/servletpath
其中 hosthome 和 port 用于与服务器建立TCP连接，而后面的URL则是用户请求服务器中的某个子容器的数据。

我们以Tomcat这个常用的容器来说明服务器是如何根据这个URL找到正确的Servlet容器。

在Tomcat中这种映射工作是由org.apache.tomcat.util.http.mapper这个专门的类去处理的。
这个类保存了Tomcat的Container容器中所有子容器的信息，当请求的Request类进入Container容器之前，
Mapper将会根据这次请求的hostname和contextPath将host和context容器设置到Request的mappingData属性中。

所以在请求进入容器之前就已经知道要访问那个子容器。

在知道要访问那个子容器后，接下来就是要执行Servlet的service方法。我们通常的做法就是不继承javax.servlet.servlet接口更为简单的HttpServlet类。
但是现在的WEB应用已经直接使用Servlet来完成用户的交互逻辑了，而是使用更为高效的MVC框架来完成这些任务。
而这些框架的入口也是Servlet的Service方法。

当Servlet从Servlet容器中移除时，也就说明Servlet的生命周期就结束了，这时Servlet的destroy方法将被调用，做一些扫尾工作。
